## 1. Scope

EPIC-0017 is a discovery epic. Its deliverables are:

1. A formal gap analysis of the Agentic SDLC Dashboard across three chapters
2. Eight implementation stories written into EPIC-0019 in `docs/RELEASE_PLAN.md`
3. EPIC-0017 marked `Done`, EPIC-0019 description updated, `docs/ID_REGISTRY.md` updated

The dashboard must become a **configurable product** ‚Äî schema + HTML extractable by any project with a config file and installer step, not just PlanVisualizer's internal pipeline.

All new implementation stories go into **EPIC-0019** (expanded from "Dashboard Cycle History" to the full effectiveness improvement set). EPIC-0019's description is updated to reflect this.

## 2. Gap Analysis

### Chapter 1 ‚Äî Schema & Data Quality

#### G1.1 ‚Äî `epics` block always empty (High)
`sdlc-status.json` has `"epics": {}` but `update-sdlc-status.js` has no `epic-start` or `epic-complete` command. The dashboard renders an epics section that is perpetually empty.

**Why it matters:** Epic-level progress is the primary unit of planning. A dashboard that can't show epic status is missing its most important signal.

**Proposed fix:** Add `epic-start` and `epic-complete` CLI commands (‚Üí US-0130).

#### G1.2 ‚Äî `"undefined"` agent entry (Medium)
When `--agent` is omitted from `agent-start` or `agent-done`, `opts.agent` is `undefined`. `ensureAgent(data, undefined)` creates a `"undefined"` key in the agents object. Visible in the dashboard as a ghost agent; corrupts task counts.

**Why it matters:** Silent corruption ‚Äî the dashboard shows an unknown agent with completed tasks, undermining trust in the metrics.

**Proposed fix:** Validate `--agent` is required on relevant commands; exit non-zero with a clear message if missing (‚Üí US-0131).

#### G1.3 ‚Äî `tasksTotal` always 0 (Medium)
`agent-done` increments `tasksCompleted` but `tasksTotal` is never set. The metric `"tasksCompleted": 14, "tasksTotal": 0` makes any progress percentage meaningless.

**Why it matters:** The dashboard's task progress bar is always undefined (divides by zero).

**Proposed fix:** `session-start --stories N` sets `storiesTotal`; task total derives from story count √ó expected tasks per story, or is declared explicitly (‚Üí US-0131).

#### G1.4 ‚Äî `coveragePercent` stays 0 (High)
The `coverage` CLI command exists and correctly sets `data.metrics.coveragePercent`. But `sdlc-status.json` shows `"coveragePercent": 0` despite Circuit logging 91.32%. Root cause: the Conductor never calls the `coverage` command ‚Äî it logs the result in the activity log but doesn't translate it into a CLI call.

**Why it matters:** Coverage is a key quality gate signal. Showing 0% when coverage is 92% actively misleads anyone reading the dashboard.

**Proposed fix:** Add explicit `coverage` call to DM_AGENT.md post-Test-phase checklist (‚Üí US-0132).

#### G1.5 ‚Äî Stale `InProgress` stories accumulate (Medium)
Stories from prior sessions remain `InProgress` indefinitely. `--force` init nukes all history. There is no command to start a fresh session while preserving historical data.

**Why it matters:** The story list becomes a graveyard of ghost stories, making it impossible to read current sprint state.

**Proposed fix:** Add `session-start` command that resets phases/stories/metrics while preserving `project` config, `agents` config, and `cycles[]` history (‚Üí US-0131).

#### G1.6 ‚Äî Log entries lack date context (Low)
`time` is stored as `HH:MM` local time. Across multi-day sessions, entries from different days appear with the same-format time and no date, making chronological ordering ambiguous.

**Why it matters:** Low ‚Äî within a single session the log is readable. But for projects that run the pipeline across multiple days, the log loses meaning.

**Proposed fix:** Change `time` to ISO 8601 timestamp; dashboard formats for display (‚Üí US-0128 AC).

### Chapter 2 ‚Äî Generalization

#### G2.1 ‚Äî `hackathon` top-level key (High)
`sdlc-status.json` contains `{ "hackathon": { "name": "SDLC Dashboard", "date": "2026-04-16", "startTime": "09:00", "endTime": "17:00" } }`. This encoding is meaningful only for a time-bounded one-day event. `init-sdlc-status.js` hardcodes this structure regardless of project type.

**Why it matters:** The key signals "this dashboard is for hackathons." Any project adopting it must manually remove or ignore this key ‚Äî not a configurable product.

**Proposed fix:** Replace `hackathon` with `project: { name, description, repoUrl, startDate }`, sourced from `agents.config.json` (‚Üí US-0127).

#### G2.2 ‚Äî Phase names hardcoded in `update-sdlc-status.js` (High)
`PHASE_DEFS` in `update-sdlc-status.js` hardcodes Blueprint/Architect/Build/Integration/Test/Polish with their agents and deliverables. Any project using different phase names gets these defaults silently applied.

**Why it matters:** Phase definitions are project-specific. Hardcoding them in a shared tool makes the tool non-reusable.

**Proposed fix:** Move `PHASE_DEFS` into `agents.config.json` as a `phases` array. `init-sdlc-status.js` seeds `sdlc-status.json.phases` from config. `update-sdlc-status.js` reads from the already-seeded JSON (‚Üí US-0129).

#### G2.3 ‚Äî Dashboard hardcodes `"My Project"` (High)
`dashboard.html` contains hardcoded project identity in four places:
- `<title>My Project ‚Äî SDLC Live Dashboard</title>`
- `<div class="header-title">My Project</div>`
- `<div class="header-subtitle">Agentic AI SDLC</div>`
- `<h3>My Project</h3>` in the about panel
- `https://github.com/yourorg/your-project` in repo links

**Why it matters:** Any adopter must search-and-replace the HTML manually. This is not a configurable product.

**Proposed fix:** Read `state.project.name` and `state.project.repoUrl` from the fetched JSON; patch all four surfaces via `patchDOM()` on each `refreshState()` tick. Log `time` display formatter added here (‚Üí US-0128).

#### G2.4 ‚Äî Config and runtime state mixed in one file (High)
`sdlc-status.json` mixes runtime state (current phase, agent statuses, log entries) with config-like data (agent names, phase definitions). Agent roles and icons live in `agents.config.json` but are not surfaced in the dashboard schema.

**Why it matters:** When `init-sdlc-status.js --force` resets the file, config data (if ever added to the state file) would be lost. Clean separation makes reasoning about each file's purpose unambiguous.

**Proposed fix:** `agents.config.json` is the config source of truth. `init-sdlc-status.js` reads it and seeds `sdlc-status.json` with the static slices (`project`, `phases`). Runtime state is never in config (‚Üí US-0127, US-0129).

#### G2.5 ‚Äî No extraction path (Medium)
Adopting the dashboard in another project requires manually copying five files and editing hardcoded strings. There is no documented procedure or installer step.

**Why it matters:** A "configurable product" claim requires a path to adoption. Without one, every adopter discovers the gaps independently.

**Proposed fix:** `docs/dashboard-extraction.md` step-by-step guide + `¬ß1: Dashboard setup` in `scripts/install.sh` (‚Üí US-0134).

### Chapter 3 ‚Äî Workflow Integration

#### G3.1 ‚Äî `phase` command never called in practice (High)
DM_AGENT.md documents calling `phase --number N --status in-progress|complete` at each transition. `sdlc-status.json` shows `currentPhase: 0` and all phases `"status": "pending"` ‚Äî the command has never been called in a real pipeline run.

**Why it matters:** The 6-phase timeline is the most prominent visual on the dashboard. Showing all phases as pending makes the entire pipeline display useless.

**Proposed fix:** DM_AGENT.md post-phase checklist gains explicit `phase` command calls (‚Üí US-0132).

#### G3.2 ‚Äî `coverage` command not in post-Test checklist (High)
DM_AGENT.md's command table lists `coverage`, but the "Post-merge checklist" and "Phase checklist" don't include an explicit call. Circuit reports coverage in the activity log message but the Conductor doesn't translate this into a `coverage` command call.

**Why it matters:** Same as G1.4 ‚Äî `coveragePercent` stays 0 in metrics.

**Proposed fix:** DM_AGENT.md Test-phase exit procedure gains: `node tools/update-sdlc-status.js coverage --agent Circuit --percent <N>` (‚Üí US-0132).

#### G3.3 ‚Äî `story-complete` doesn't auto-idle the assigned agent (Medium)
After `story-complete`, the story's `assignedAgent` field is set but the agent's status remains at whatever it last was. The Conductor must separately call `agent-done` to idle the agent ‚Äî easy to forget, leaves agents visually "active" on completed work.

**Why it matters:** Agent status cards showing "active" on completed stories confuse the live view.

**Proposed fix:** `story-complete` handler auto-calls the agent-idle logic for `story.assignedAgent` if set (‚Üí US-0132).

#### G3.4 ‚Äî No `session-start` / `session-reset` command (High)
Starting a new pipeline session has two bad options: leave stale stories (ghost data) or use `--force` init (nukes all history). There is no clean middle path.

**Why it matters:** Every new epic/sprint starts with a corrupted dashboard showing the previous session's ghost stories.

**Proposed fix:** `session-start` command resets phases/stories/metrics, preserves `project`, `agents`, and `cycles[]` (‚Üí US-0131).

#### G3.5 ‚Äî `storiesTotal` grows dynamically, not declared (Medium)
`story-start` sets `storiesTotal = Math.max(storiesTotal, Object.keys(data.stories).length)`. This means `storiesTotal` grows as stories are added rather than being declared at session start. The phase progress bar's denominator is always shifting.

**Why it matters:** A progress bar whose total changes mid-run is misleading.

**Proposed fix:** `session-start --stories N` declares `storiesTotal` upfront (‚Üí US-0131).

#### G3.6 ‚Äî Bug metrics never updated (Low)
`metrics.bugsOpen` and `bugsFixed` exist in the schema but no CLI command updates them. Bugs filed during the pipeline have no dashboard representation.

**Why it matters:** Low ‚Äî bug counts are secondary to story/coverage metrics in the live view. But the schema misleads: it implies tracking that doesn't happen.

**Proposed fix:** Add `bug-open` and `bug-fix` commands. `bugsOpen` has a `Math.max(0, ...)` floor guard to prevent negative values from unpaired calls (‚Üí US-0132).

### US-0127 ‚Äî Schema Generalization *(Estimate: S)*

**As a** project adopter, **I want** `sdlc-status.json` to use a `project` config block instead of `hackathon`, **so that** the schema is meaningful for any ongoing project, not just a one-day event.

**Gaps:** G2.1, G2.4

**Acceptance Criteria:**
- `agents.config.json` gains a `project` section with `name`, `description`, `repoUrl`, `startDate` fields
- `init-sdlc-status.js` reads `config.project` and writes it as `sdlc-status.json.project`
- `sdlc-status.json` no longer contains a `hackathon` key
- All `update-sdlc-status.js` handlers preserve `data.project` on every mutation (never overwrite)
- Existing `sdlc-status.json` migration: `hackathon` ‚Üí `project` with field mapping (`name` preserved, `date` ‚Üí `startDate`, `startTime`/`endTime` dropped)
- Unit tests cover init, mutation preservation, and migration path

---

### US-0128 ‚Äî Dashboard Dynamic Project Identity *(Estimate: S)*

**As a** project adopter, **I want** the dashboard to read its title and repo links from `sdlc-status.json`, **so that** I don't need to edit HTML to adopt it.

**Gaps:** G2.3, G1.6

**Acceptance Criteria:**
- `<title>` tag is updated at page load from `state.project.name` on first successful fetch
- `header-title` element is patched to `state.project.name` on each `refreshState()` tick
- `header-subtitle` element is patched to `state.project.description` on each tick
- About panel `<h3>` and GitHub repo links read from `state.project.name` / `state.project.repoUrl`
- Log `time` field changed to ISO 8601 in `update-sdlc-status.js`; dashboard formats it as `HH:MM` for display (existing `HH:MM` strings rendered as-is for backwards compatibility)
- All hardcoded `"My Project"` and `"yourorg/your-project"` strings removed from HTML
- Unit tests cover `patchDOM()` with a mock state containing `project` fields

---

### US-0129 ‚Äî Phase Config Externalization *(Estimate: M)*

**As a** project adopter, **I want** phase names and agents to come from `agents.config.json`, **so that** my project's phases appear correctly without editing `update-sdlc-status.js`.

**Gaps:** G2.2

**Acceptance Criteria:**
- `agents.config.json` gains a `phases` array: `[{ name, agents[], deliverables[] }]`
- `init-sdlc-status.js` seeds `sdlc-status.json.phases` from `config.phases` at init time (with `status: "pending"`, `startedAt: null`, `completedAt: null`)
- `update-sdlc-status.js` `phase` handler reads phase definitions from `data.phases` (already seeded) rather than the internal `PHASE_DEFS` constant
- `PHASE_DEFS` constant is removed from `update-sdlc-status.js`
- `--force` init regenerates phases from config
- Unit tests cover init seeding and phase command with config-driven names

---

### US-0130 ‚Äî Epic Lifecycle Commands *(Estimate: M)*

**As a** Conductor agent, **I want** `epic-start` and `epic-complete` CLI commands, **so that** the dashboard shows epic-level progress alongside story progress.

**Gaps:** G1.1

**Acceptance Criteria:**
- `epic-start --epic EPIC-XXXX --name "..." --stories N` creates `epics[id]` entry with `{ name, status: "in-progress", startedAt, completedAt: null, storiesCompleted: 0, storiesTotal: N }`
- `epic-complete --epic EPIC-XXXX` sets `status: "complete"` and `completedAt` on `epics[id]`
- `story-complete` with `--epic EPIC-XXXX` increments `epics[id].storiesCompleted` if the epic entry exists
- Dashboard renders a compact epic-progress strip above the story list: epic name, `storiesCompleted/storiesTotal`, status badge
- DM_AGENT.md updated: Conductor calls `epic-start` at session start and `epic-complete` after all stories in the epic merge
- Unit tests cover `epic-start`, `epic-complete`, and `story-complete` epic increment

---

### US-0131 ‚Äî Session Reset & CLI Corrections *(Estimate: M)*

**As a** Conductor agent, **I want** a `session-start` command and validated required flags, **so that** each new pipeline session begins with clean state and ghost data cannot accumulate.

**Gaps:** G1.2, G1.3, G1.5, G3.4, G3.5

**Acceptance Criteria:**
- `session-start --stories N` resets `phases` (all `pending`), `stories` (empty), `metrics` (all 0 except `storiesTotal: N`), and `currentPhase: 0`; preserves `project`, `agents`, `epics`, `cycles`, and `log`. Note: `storiesTotal` here is `metrics.storiesTotal` (the dashboard's overall progress bar denominator) ‚Äî distinct from `epics[id].storiesTotal` which is set by `epic-start`
- `agent-start` and `agent-done` exit non-zero with message `"--agent is required"` if `opts.agent` is undefined or the string `"undefined"`
- `story-start` and `story-complete` exit non-zero with message `"--story is required"` if `opts.story` is undefined
- `storiesTotal` is set from `--stories N` on `session-start`; `story-start` no longer modifies `storiesTotal`
- DM_AGENT.md updated: Conductor calls `session-start --stories N` before spawning the first story in any new epic
- Unit tests cover session-start reset behaviour, flag validation, and `storiesTotal` initialization

---

### US-0132 ‚Äî Coverage & Bug Metrics Wiring *(Estimate: S)*

**As a** Conductor agent, **I want** coverage, bug, and phase transition calls to be part of the standard pipeline checklist, **so that** dashboard metrics reflect real execution state.

**Gaps:** G1.4, G3.1, G3.2, G3.3, G3.6

**Acceptance Criteria:**
- `bug-open --story US-XXXX` increments `metrics.bugsOpen`; `bugsOpen` has a `Math.max(0, ...)` floor guard
- `bug-fix --story US-XXXX` decrements `metrics.bugsOpen` (floored at 0), increments `metrics.bugsFixed`
- `story-complete` handler auto-idles the story's `assignedAgent` (sets `status: "idle"`, `currentTask: null`) if the agent exists in `data.agents`
- DM_AGENT.md post-phase checklist gains:
  - `phase --number N --status complete` at the end of each phase
  - `phase --number N+1 --status in-progress` at the start of each phase
  - `coverage --agent Circuit --percent <N>` after Circuit reports results in Test phase
- DM_AGENT.md post-story checklist gains `bug-open`/`bug-fix` calls as appropriate
- Unit tests cover `bug-open`, `bug-fix` (including floor guard), and `story-complete` agent auto-idle

---

### US-0133 ‚Äî Cycle History *(Estimate: L)*

**As a** Conductor reviewing pipeline performance, **I want** completed cycle snapshots in the dashboard, **so that** I can see lap history, average cycle time, and trend data across epics.

**Gaps:** EPIC-0019 original scope

**Acceptance Criteria:**
- `sdlc-status.json` gains a `cycles: []` array at the root level
- `cycle-complete` command snapshots current metrics (`storiesCompleted`, `testsPassed`, `coveragePercent`, `bugsFixed`, `phaseDurations`) into a new `cycles[]` entry with `{ id, completedAt, ... }`, then applies the same reset mutations as `session-start` (via a shared internal `resetSession(data)` function ‚Äî not a recursive CLI call) so no separate `session-start` call is needed
- `phaseDurations` is computed from `startedAt`/`completedAt` on each phase entry (seconds)
- Dashboard renders a lap-history strip below the pipeline: last 10 cycles as compact cards showing cycle number, completion time, stories, coverage
- Dashboard renders an aggregate telemetry row: avg cycle time, cycles today, overall success rate (cycles with 0 failed tests / total cycles)
- A cycle-completion animation plays when `cycles.length` increases between two `refreshState()` ticks
- DM_AGENT.md updated: Conductor calls `cycle-complete` after all stories in an epic merge (replaces the prior `session-start` guidance)
- Unit tests cover `cycle-complete` snapshot, reset side-effect, `phaseDurations` computation, and lap-strip rendering

---

### US-0134 ‚Äî Dashboard Extraction Guide *(Estimate: S)*

**As a** developer adopting PlanVisualizer's dashboard for a new project, **I want** a documented extraction procedure and an install step, **so that** I can set up the dashboard without manual search-and-replace.

**Gaps:** G2.5

**Acceptance Criteria:**
- `docs/dashboard-extraction.md` documents the step-by-step procedure: copy files, populate `agents.config.json` `project` and `phases` sections, run `init-sdlc-status.js`, open `dashboard.html`
- `scripts/install.sh` gains a `¬ß1: Dashboard setup` section that: copies `dashboard.html`, `docs/sdlc-status.json` (template from `init-sdlc-status.js --force`), `tools/update-sdlc-status.js`, `tools/init-sdlc-status.js` to a user-specified target directory; prompts for project name and repo URL; writes them into `agents.config.json`
- Install script `¬ß1` is skipped (with a note) if `docs/dashboard.html` already exists in the target directory
- `docs/dashboard-extraction.md` is linked from `README.md` under a "Dashboard" section
- No test requirement (documentation + shell script only)
